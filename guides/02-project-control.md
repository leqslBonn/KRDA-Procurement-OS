# คู่มือ 02 — คุมงานจ้าง (Owner: SOW → ตรวจรับ → จ่ายเงิน)

> **งานนี้คืออะไร:** เลือก vendor ได้แล้ว จ้างให้เขาทำ (สร้างระบบ/พัฒนา) แล้วคุณเป็น **เจ้าของงาน (owner)** ต้องคุมให้ได้งานตรง จ่ายเงินถูกจังหวะ ไม่โดนเอาเปรียบ
> **ใช้เมื่อไหร่:** ตั้งแต่หลังเลือก vendor จนงานเสร็จ+หมดประกัน
> **โปรแกรม:** 🚀 SOW · 📑 เช็คสัญญา · 💰 งวดจ่ายเงิน · ✅ ตรวจรับงาน

---

## 🎯 หัวใจของงานนี้ (จำไว้ 1 ประโยค)
> **"จ่ายตามงานที่ตรวจรับผ่าน ไม่จ่ายตามคำสัญญา"** — เงินคืออำนาจคุมเดียวที่คุณมีเหนือ vendor

## ⏱️ ลำดับงานทั้งหมด
```
① เขียน SOW (ขอบเขตงาน) → ② ทำสัญญา (เช็คเงื่อนไข) → ③ แบ่งงานเป็นงวด
→ ④ vendor ทำ → ⑤ ตรวจรับแต่ละงวด → ⑥ จ่ายเมื่อรับผ่าน → ⑦ จบงาน+ประกัน → คืนเงินประกัน
```

---

## 🟦 STEP 1 — เขียน SOW (ขอบเขตงาน) *ทำก่อนเซ็นสัญญาเสมอ*

**ทำไมสำคัญ:** ถ้าขอบเขตไม่ชัด → vendor จะเถียงทีหลังว่า "งานนี้อยู่นอกขอบเขต ต้องจ่ายเพิ่ม" (scope creep)

**วิธีทำ — เปิดแท็บ 🚀 SOW กรอก:**
1. **ชื่อโครงการ** + **วัตถุประสงค์** (ผลลัพธ์ที่ต้องการ)
2. **อยู่ในขอบเขต (in-scope)** — งานที่ vendor ต้องทำ เขียนให้ครบ 1 บรรทัด/ข้อ
3. **นอกขอบเขต (out-scope)** — เขียนสิ่งที่ **ไม่** รวม ให้ชัด (อันนี้แหละที่กันงานบานปลาย)
4. **สิ่งที่ต้องส่งมอบ** — ระบบใช้งานได้ + **source code + เอกสาร + ส่งมอบข้อมูล** (ไม่ใช่แค่ "ใช้ได้")
5. **งวดงาน + % จ่าย** — แบ่งเป็นเฟส เช่น:
   - M1 ออกแบบ/อนุมัติแบบ = 20%
   - M2 สร้าง/POC = 30%
   - M3 เชื่อมระบบ = 30%
   - M4 ทดสอบ/ใช้จริง = 20%
6. กด **สร้างเอกสาร SOW** → พิมพ์/แนบท้ายสัญญา

---

## 🟩 STEP 2 — เช็คเงื่อนไขสัญญา *ก่อนเซ็น*

**เปิดแท็บ 📑 เช็คสัญญา → ติ๊กให้ครบ** โดยเฉพาะข้อ **critical**:
- ☑️ ขอบเขต = SOW แนบท้าย
- ☑️ **KRDA เป็นเจ้าของ IP/ผลงาน + source code**
- ☑️ **KRDA เป็นเจ้าของ data + export ได้**
- ☑️ **จ่ายตามงวด** (ไม่จ่ายล่วงหน้าก้อนใหญ่)
- ☑️ **Source-code escrow** (เผื่อ vendor เจ๊ง)

ระบบจะบอก **"พร้อมเซ็น" หรือ "ยังไม่พร้อม (ขาดอะไร)"**

📌 **ถ้ายังขาด critical → ยังไม่เซ็น** ให้กลับไปเจรจา (ดู [คู่มือ 03 ส่วนเจรจา](03-buying-and-negotiation.md))

⚠️ **สัญญา = ผูกมัด (commitment)** → คุณเตรียมได้ แต่ **ผู้มีอำนาจเป็นคนเซ็น** (ดู [คู่มือ 05](05-documents-approval.md))

---

## 🟨 STEP 3 — ตั้งงวดจ่ายเงิน (ติดตามทั้งโครงการ)

**เปิดแท็บ 💰 งวดจ่ายเงิน:**
1. ใส่ **มูลค่าสัญญารวม** (บาท)
2. ใส่ **% กันเงินประกัน (retention)** — แนะนำ 10–20%
3. ตารางงวด: ใส่ชื่องวด + % จ่าย (รวมทุกงวด = 100%)
4. พองานงวดไหนตรวจรับผ่าน → เปลี่ยนสถานะเป็น **"รับแล้ว"**
5. ระบบคำนวณ: **จ่ายแล้วเท่าไหร่ · กันประกันไว้เท่าไหร่ · เหลือเท่าไหร่**

---

## 🟧 STEP 4 — ตรวจรับงานแต่ละงวด *(ทำทุกครั้งที่ vendor ส่งงาน)*

**เปิดแท็บ ✅ ตรวจรับงาน:**
1. ใส่ชื่อโครงการ + งวด (เช่น M3)
2. **เกณฑ์รับงาน** — ใส่เกณฑ์จาก SOW แล้วติ๊กว่าผ่าน/ไม่ผ่านทีละข้อ
3. **Punch list** — จดของที่ยังไม่ผ่าน (ระบุ major/minor + สถานะ)
4. กด **ประเมินผล** → ระบบบอก:
   - ✅ **รับได้** (เกณฑ์ผ่าน + ไม่มี punch major ค้าง) → จ่ายงวดนี้ (กันประกันไว้)
   - ❌ **ยังรับไม่ได้** → **ไม่จ่าย** ให้ vendor แก้ punch list ก่อน

📌 **"ปฏิเสธ" คือไพ่แข็งที่สุดของคุณ** — อย่ารับงานที่ยังไม่ครบเพราะเกรงใจ

---

## 🟪 STEP 5 — จบงาน + ประกัน
1. งวดสุดท้าย (UAT/ใช้จริง) ผ่าน → รับมอบงานเต็ม
2. ได้ครบ: ระบบ + source code + เอกสาร + ข้อมูล
3. เข้าช่วง **warranty** (แก้บั๊กฟรี N เดือน)
4. หมดประกัน + ไม่มีปัญหา → **คืนเงินประกัน (retention)**

---

## ⚠️ กับดักที่พบบ่อย
| ❌ อย่า | ✅ ให้ทำ |
|--------|---------|
| จ่ายล่วงหน้า 50% ก่อนเห็นงาน | จ่ายตามงวดที่ตรวจรับผ่าน |
| เริ่มงานก่อนขอบเขตชัด | เขียน SOW ให้ครบก่อน |
| รับงานเพราะเกรงใจ ทั้งที่ยังไม่ครบ | ทำ punch list ให้แก้ก่อน |
| เซ็นสัญญาที่ vendor ยึด data/IP | เช็คให้ครบ critical ก่อนเซ็น |
| ยอมให้เพิ่มงานฟรี | เปลี่ยนขอบเขต = change order เขียน+ตีราคา |

---

## ✅ Checklist ทั้งโครงการ
- [ ] เขียน SOW + เกณฑ์รับงาน (วัดได้) ก่อนเลือก/เซ็น
- [ ] เช็คสัญญาครบ critical (IP/data/งวด/retention/escrow) ก่อนเซ็น
- [ ] แบ่งงวด รวม 100% + กันเงินประกัน
- [ ] ตรวจรับทุกงวด → จ่ายเฉพาะที่ผ่าน
- [ ] ทุกการจ่าย ผู้มีอำนาจอนุมัติ
- [ ] คืน retention ตอนหมดประกัน
